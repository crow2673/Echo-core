# The One-Line Bug That Killed My AI Content Pipeline for Two Weeks

I've always been fascinated by the power of AI, especially how it can automate content creation. Being a trades worker turned AI builder, I’ve learned a lot by doing, and sometimes, I’ve learned the hard way. Recently, I ran into a frustrating issue with my AI content pipeline that took two weeks to resolve. The culprit? A one-line bug in `is_ollama_idle()`.

## Setting Up the Pipeline

I use a Ryzen 9 5900X with an RTX 3060 12GB, 32GB of RAM, and Ubuntu Linux for my local AI setup. I run qwen2.5:32b via Ollama directly on my system without Docker or cloud dependencies. The setup is straightforward, and I installed Ollama using:

```bash
curl -fsSL https://ollama.com/install.sh | sh
```

## The Pipeline Code

The core of my pipeline is a Python script named `article_pipeline.py`, which handles fetching drafts, processing them through Ollama, and then pushing the generated content to my blog. Here's a simplified version of the relevant part of the code:

```python
import urllib.request
import json

def is_ollama_idle():
    url = "http://localhost:11434/api/ps"
    with urllib.request.urlopen(url) as response:
        data = json.loads(response.read().decode())
        models = data.get("models", [])
        return len(models) == 0

def process_draft(draft):
    if not is_ollama_idle():
        print("Ollama is not idle, skipping draft")
        return
    # Processing logic here
    print(f"Processed draft: {draft}")

# Example usage
drafts = ["draft1.md", "draft2.md"]
for draft in drafts:
    process_draft(draft)
```

The `is_ollama_idle()` function checks if Ollama is idle by querying the `/api/ps` endpoint and parsing the JSON response. The function returns `True` if no models are loaded (i.e., `len(models) == 0`), indicating that Ollama is idle and ready to process a new request.

## The Bug Strikes

Everything seemed to work fine initially, but after a few days, I noticed that my pipeline was failing to process drafts. The drafts would sit in the queue marked as "failed." I couldn't figure out what was going wrong until I started logging the output of `is_ollama_idle()`.

```bash
$ python3 article_pipeline.py
Ollama is not idle, skipping draft
Ollama is not idle, skipping draft
```

This was confusing because I knew Ollama was running and should have been idle. Digging deeper, I realized that `is_ollama_idle()` was returning `False` because even a single warm model in VRAM (like Echo or gai) made the function return `False`. This meant that the pipeline almost never got past the idle check.

## The Fix

The fix was straightforward but took me two weeks to identify and implement. Instead of checking if no models were loaded, I simply needed to check if Ollama was reachable. This change made the pipeline much more robust.

Here's the corrected `is_ollama_idle()` function:

```python
def is_ollama_idle():
    url = "http://localhost:11434/api/ps"
    try:
        with urllib.request.urlopen(url) as response:
            return True
    except Exception as e:
        print(f"Failed to reach Ollama: {e}")
        return False
```

With this change, the pipeline started working again, and drafts were processed without any issues.

## What's Still Broken

While the pipeline is now functional, I still need to improve how I handle retries and failures. Right now, if Ollama is not reachable, the script fails silently. I plan to add more robust error handling and logging to catch these issues early.

Also, I need to explore better ways to manage multiple drafts in the queue. Currently, the pipeline processes one draft at a time, which can be inefficient. I'm considering implementing a more sophisticated queue system to handle multiple drafts in parallel.
